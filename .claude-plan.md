# Chat Panel UI Deep Dive - Agent Generation Progress

## Overview
The chat panel is a **LiveComponent** that displays a vertical timeline of agent execution events. It's hosted in the editor LiveView (`edit.ex`) and receives real-time updates via PubSub messages showing agent progress, tool execution, streaming text, and proposed edits.

## Event Types Rendered

The component groups and renders these event types (lines 537-574):

### 1. `:message` Events
**Renders**: `render_message_step/1` (lines 203-236)
- User or assistant messages
- Fields: `role` ("user" or "assistant"), `content`, `timestamp`
- UI: Styled boxes with role icon, timestamp, message text
- Timeline node: Blue border for user, muted for assistant

### 2. `:tool_call` Events
**Renders**: `render_tool_step/1` (lines 240-332) 
- Agent calling a tool (generate_code, compile_code, format_code, lint_code, run_tests, etc.)
- Fields: `tool`, `args`, `timestamp`, `id`, `tool_duration_ms`
- UI: Collapsible step with tool icon, name, status badge (pulsing while executing)
- Expandable detail shows timestamps, input code, test code

### 3. `:tool_result` Events
**Renders**: `render_tool_step/1` (when paired with call) or `render_standalone_result/1` (lines 346-370)
- Tool execution result
- Fields: `tool`, `success`, `content`, `timestamp`, `tool_duration_ms`
- UI: Shows success/failure icon, formatted output (code block or text)
- Duration displayed in expanded section

### 4. `:status` Events
**Renders**: `render_status_step/1` (lines 374-381)
- Generic status messages
- Fields: `content`
- UI: Small italic text with faint dot marker

### 5. `:run` Summary
**Renders**: `render_run_summary/1` (lines 137-188)
- Displayed at END of timeline when run completes (`@run && !@loading`)
- Shows: Run type, status badge, model, timestamps, duration, token counts, costs
- UI: Chart icon, status colors (running=blue animate, completed=green, failed=red)

## Event Grouping Logic

The `group_events/1` function (lines 536-565) uses `Enum.chunk_while/4` to:
1. Pair `:tool_call` with immediately following `:tool_result` → creates `:tool_group` tuple
2. Handle orphaned tool calls (no result yet) → creates `:tool_call` tuple
3. Handle orphaned results → creates `:tool_result` tuple
4. Wrap other events with their type tag

**Output**: List of tuples:
```elixir
[
  {:message, event},
  {:tool_group, call, result},
  {:tool_call, call},
  {:tool_result, result},
  {:status, event}
]
```

## PubSub Subscription & Message Flow

### Subscription Setup (edit.ex, lines 54-55)
```elixir
# Subscribe to API topic for agent_run_started
Phoenix.PubSub.subscribe(Blackboex.PubSub, "api:#{api.id}")
```

When agent starts (`:agent_run_started` received):
```elixir
# Lines 1822-1840
def handle_info({:agent_run_started, %{run_id: run_id, run_type: _type}}, socket) do
  # Unsubscribe from old run topic if exists
  if old_run = socket.assigns.current_run_id do
    Phoenix.PubSub.unsubscribe(Blackboex.PubSub, "run:#{old_run}")
  end
  
  # Subscribe to this run's topic for events
  Phoenix.PubSub.subscribe(Blackboex.PubSub, "run:#{run_id}")
  
  # Load run struct
  run = AgentConversations.get_run(run_id)
  
  {:noreply, assign(socket, current_run_id: run_id, current_run: run, chat_loading: true, active_tab: "chat")}
end
```

### Message Handlers (lines 1843-2006)

| Handler | Event | Effect |
|---------|-------|--------|
| `handle_info({:agent_streaming, ...})` (1843) | `{:agent_streaming, %{delta: delta}}` | Appends `delta` to `@streaming_tokens`, re-renders |
| `handle_info({:agent_action, ...})` (1854) | `{:agent_action, %{tool: tool_name, args: args}}` | Creates `:tool_call` event, updates `pipeline_status` |
| `handle_info({:agent_action, ...})` (1871) | `{:agent_action, %{tool: tool_name}}` | Updates `pipeline_status` only |
| `handle_info({:tool_started, ...})` (1876) | `{:tool_started, %{tool: tool_name}}` | Updates `pipeline_status` |
| `handle_info({:tool_result, ...})` (1881) | `{:tool_result, %{tool: tool_name, success: bool, ...}}` | Creates `:tool_result` event, applies to editor if relevant |
| `handle_info({:agent_completed, ...})` (1909) | `{:agent_completed, %{code: ..., run_id: ...}}` | Unsubscribes from run, refreshes API, shows diff or accepts directly |
| `handle_info({:agent_failed, ...})` (1959) | `{:agent_failed, %{error: ..., run_id: ...}}` | Unsubscribes, shows error flash |
| `handle_info({:agent_message, ...})` (1988) | `{:agent_message, %{role: "assistant", content: ...}}` | Creates `:message` event |
| `handle_info({:agent_started, ...})` (2006) | `{:agent_started, ...}` | No-op |

## Real-time Update Flow

```
PubSub Message
    ↓
handle_info/2 in edit.ex
    ↓
Create/Update assigns (agent_events, streaming_tokens, pipeline_status, etc.)
    ↓
LiveView re-renders (Phoenix optimizes to send only changed assigns)
    ↓
ChatPanel component receives updated assigns
    ↓
group_events/1 re-groups events
    ↓
Renders timeline with new event or updated streaming text
    ↓
JS updates DOM (smooth scrolling to bottom happens automatically via overflow-y-auto)
```

3. **Diff visualization** — Uses colored diff lines (green `+`, red `-`, plain ` `)
4. **Validation badges** — Show compile/format/credo/test status with color codes

8. **Run summary timing** — Shows start→end with formatted duration
9. **Cost tracking** — Displays input/output tokens and cost in cents

## Summary

The chat panel is a **purely server-driven, event-based timeline UI** that:
- Receives PubSub messages from the agent pipeline
- Accumulates events into `@agent_events` list
- Groups events intelligently (pairing tool calls with results)
- Renders a vertical timeline with collapsible tool steps
- Shows real-time streaming text with a blinking cursor
- Displays proposed code diffs for review before acceptance
- Includes validation status badges (compile, format, lint, tests)
- Shows a final run summary with metrics and costs

No custom JS streaming handlers are needed — all updates flow through LiveView's reactive assign system and PubSub message handling.
