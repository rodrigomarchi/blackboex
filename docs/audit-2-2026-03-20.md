# Auditoria de Producao - BlackBoex

> **Data:** 2026-03-20
> **Suite de testes:** 787 tests, 0 failures (552 domain + 235 web)
> **Escopo:** Auditoria profunda para validacao de production-readiness
> **Metodologia:** Leitura de codigo, nao apenas verificacao de existencia

---

## Resumo Executivo

A plataforma BlackBoex tem **fundamentos de seguranca solidos**: autenticacao bcrypt, RBAC via LetMe, isolamento multi-tenant, API keys com hash SHA-256 e comparacao timing-safe, CSRF ativo, sandbox com AST validation abrangente.

A auditoria identificou 5 problemas criticos, 6 importantes, e 8 menores. **Todos os 19 problemas foram corrigidos em 2026-03-20:** billing enforcement com usage events para invocations, sandbox bloqueando HTTP clients, erros genericos ao usuario, webhook atomico, SSL habilitado, effective_plan com subscription check, TOCTOU protegido, Dockerfile para deploy, graceful shutdown, e 25 novos testes.

## Problemas Criticos (MUST FIX)

### C1. UsageEvent nunca gravado para API invocations — billing bypass

**Arquivo:** `apps/blackboex_web/lib/blackboex_web/plugs/dynamic_api_router.ex:270`

`log_request` chama `Analytics.log_invocation()` (tabela `invocation_logs`, apenas analytics). **Nao chama** `Billing.record_usage_event()` (tabela `usage_events`, usada pelo enforcement).

`Enforcement.check_limit(org, :api_invocation)` consulta `Billing.sum_monthly_usage(org_id, "api_invocation")` que conta registros em `usage_events`. Como nenhum registro e inserido, o count e sempre 0.

**Impacto:** Usuarios podem fazer invocacoes ilimitadas de APIs publicadas. O enforcement existe no codigo mas nunca e acionado porque nao ha dados para contar.

**Fix:** Adicionar `Billing.record_usage_event(%{organization_id: api.organization_id, event_type: "api_invocation", metadata: %{api_id: api.id}})` em `log_request` ou no pipeline antes da execucao.

---

### C2. Sandbox nao bloqueia bibliotecas HTTP (Req, Finch, Mint)

**Arquivo:** `apps/blackboex/lib/blackboex/code_gen/ast_validator.ex:9-31`

O AST validator bloqueia `File`, `System`, `IO`, `:gen_tcp`, `:httpc`, etc. Mas **nao bloqueia**:
- `Req` (HTTP client usado pelo proprio projeto)
- `:finch` (backend HTTP do Req)
- `:mint` (HTTP client de baixo nivel)
- `HTTPoison` (se disponivel)

Essas bibliotecas estao nas dependencias do projeto e sao acessiveis pelo codigo do usuario.

**Impacto:** Codigo de usuario pode fazer requests HTTP para servicos internos (SSRF), exfiltrar dados, ou atacar outros sistemas.

**Fix:** Adicionar ao `@blocked_modules`: `Req, Req.Request, HTTPoison` e ao `@blocked_erlang_modules`: `:finch, :mint, :hackney`.

---

### C3. Exception.message exposta ao usuario em respostas de API

**Arquivo:** `apps/blackboex/lib/blackboex/code_gen/module_builder.ex:33,66,131`

```elixir
rescue
  e -> send_resp(conn, 500, Jason.encode!(%{error: "handler error", detail: Exception.message(e)}))
```

Em todos os 3 templates (computation, crud, webhook), erros do handler do usuario sao enviados ao cliente com `Exception.message(e)`.

**Impacto:** Pode expor caminhos de arquivos, nomes de modulos, detalhes de SQL, ou estado interno do sistema.

**Fix:** Logar o erro completo e retornar mensagem generica:
```elixir
rescue
  e ->
    Logger.error("Handler error: #{Exception.message(e)}")
    send_resp(conn, 500, Jason.encode!(%{error: "handler error"}))
```

### C4. Webhook idempotency com race condition TOCTOU

**Arquivo:** `apps/blackboex/lib/blackboex/billing/webhook_handler.ex:15-29`

```elixir
if already_processed?(event_id) do     # CHECK
  {:error, :already_processed}
else
  case handle_event(event_type, payload) do
    :ok -> mark_processed(event_id, event_type)  # MARK (gap entre check e mark)
```

Dois webhooks concorrentes com mesmo `event_id` podem ambos passar o check e executar a logica de billing.

**Impacto:** Subscriptions duplicadas, double-credit de planos.

**Fix:** Usar insert atomico com unique constraint:
```elixir
case Repo.insert(%ProcessedEvent{event_id: event_id, ...}, on_conflict: :nothing) do
  {:ok, %{id: nil}} -> {:error, :already_processed}  # conflict
  {:ok, _} -> handle_event(event_type, payload)       # first time
end
```

---

### C5. Database SSL desabilitado em producao

**Arquivo:** `config/runtime.exs:61`

```elixir
# ssl: true,
```

Linha comentada no bloco de configuracao de producao.

**Impacto:** Conexoes ao banco em producao sao nao-encriptadas. Man-in-the-middle pode interceptar credenciais e dados.

**Fix:** Descomentar `ssl: true` e configurar validacao de certificado se necessario.

## Problemas Importantes (SHOULD FIX)

### I1. Health check redirecionado para HTTPS

**Arquivo:** `config/prod.exs:15-20`

`force_ssl` esta ativo mas nao exclui paths de health check (`/health/live`, `/health/ready`, `/health/startup`). Load balancers que fazem health check via HTTP vao receber redirect 301.

**Fix:** Adicionar `exclude: [paths: ["/health"]]` ao `force_ssl`.

---

### I2. Plug.Parsers sem limite de body size

**Arquivo:** `apps/blackboex_web/lib/blackboex_web/endpoint.ex:49-53`

```elixir
plug Plug.Parsers,
  parsers: [:urlencoded, :multipart, :json],
  pass: ["*/*"],
  body_reader: {BlackboexWeb.Plugs.CacheBodyReader, :read_body, []},
  json_decoder: Phoenix.json_library()
```

Sem `:length` configurado. O default do Plug e 8MB, que pode ser alto demais para APIs JSON.

**Fix:** Adicionar `length: 1_000_000` (1MB) ou valor apropriado.

---

### I3. Plano da org nao valida status da subscription

**Arquivo:** `apps/blackboex/lib/blackboex/billing/enforcement.ex`

`check_limit` usa `org.plan` diretamente. Se o webhook de cancelamento falhar, a org mantem o plano pro/enterprise sem subscription ativa.

**Fix:** Implementar `effective_plan(org)` que verifica subscription status antes de retornar o plano.

---

### I4. TOCTOU na criacao de APIs (limite de plano)

**Arquivo:** `apps/blackboex/lib/blackboex/apis.ex:30-34`

```elixir
with :ok <- check_api_limit(org_id) do
  %Api{} |> Api.changeset(attrs) |> Repo.insert()
```

Requests concorrentes podem ambos passar o check e exceder o limite.

**Fix:** Usar `Ecto.Multi` com check dentro da transacao, ou aceitar como risco baixo para MVP.

---

### I5. Admin LiveViews sem nenhum teste

6 LiveViews do admin panel (Dashboard, Organizations, APIs, Users, Subscriptions, Audit Logs) tem **zero cobertura de testes**. A autorizacao depende do plug `RequirePlatformAdmin` no router, que nunca foi testado.

**Fix:** Criar testes que verificam: non-admin recebe 403/redirect, admin acessa, e operacoes CRUD funcionam.

---

### I6. Edit LiveView com ~23% de cobertura de eventos

56 `handle_event` clauses no edit.ex, apenas ~13 testados. Eventos criticos sem teste:
- `send_request` (executa API)
- `send_chat` (envia para LLM)
- `run_tests` (executa codigo gerado)
- `quick_test` (executa API rapido)
- Todos os eventos do request builder (params, headers, URL)

**Fix:** Adicionar testes para eventos de maior risco.

## Problemas Menores (NICE TO FIX)

### M1. IP spoofing no rate limiter

`conn.remote_ip` sem configuracao verificada de `Plug.RemoteIp`. Se atras de proxy, IPs podem ser spoofados via `X-Forwarded-For`.

### M2. Sem guard de idempotencia em save/compile

Clicks rapidos em "Save" ou "Save & Compile" podem criar versoes duplicadas. Outros eventos (generate_tests, run_tests, generate_docs) ja tem guards.

### M3. test_params/test_headers sem limite no LiveView

Usuario pode adicionar params/headers infinitamente, crescendo a memoria do processo.

### M4. Campos JSONB sem limite de tamanho

`test_requests.headers`, `apis.param_schema`, etc. podem ter tamanho arbitrario. Adicionar validacao de tamanho maximo.

### M5. Index faltando em usage_events

Falta index composto `(organization_id, event_type, inserted_at)` para a query `count_usage_events_today`.

### M6. Payload de webhook sem validacao estrita

Campos como `organization_id`, `customer`, `subscription` nao sao validados antes do uso.

### M7. Sem Dockerfile ou release config

Deployment precisa de container config para producao.

### M8. Sem graceful shutdown para modulos dinamicos

APIs em execucao podem crashar no shutdown. Considerar drain de requests.

## Seguranca — Achados Positivos

| Area | Status | Detalhes |
|------|--------|---------|
| Autenticacao | SOLIDA | Bcrypt, magic links com expiry 15min, timing-attack protection |
| Sessao | SOLIDA | Token SHA-256 no DB, rotacao apos 7d, validade 14d, session fixation protection |
| RBAC | SOLIDO | LetMe com 3 roles (owner/admin/member), verificacao em mount de LiveViews |

| API Keys | SOLIDO | Hash SHA-256, Plug.Crypto.secure_compare, rate limit por key |
| AST Validation | FORTE | 12 modulos Elixir + 12 erlang bloqueados (incl. HTTP clients), apply/spawn/send bloqueados, max 500 atoms |
| Sandbox | FORTE | Task.Supervisor isolado, timeout, max_heap com kill:true |
| CSRF | ATIVO | protect_from_forgery, bypass correto em /api |
| Webhook Stripe | SEGURO | Signature verification, raw body caching |
| SQL Injection | SEGURO | Todas queries parametrizadas via Ecto |
| XSS | SEGURO | Nenhum uso de raw(), HEEx auto-escapa |
| Secrets | SEGURO | Todos via env vars, raise se ausente em prod |
| Sensitive Data | SEGURO | Password redact:true, headers redacted em logs |
| Audit Trail | SEGURO | ExAudit + custom audit log, scoped por org |
