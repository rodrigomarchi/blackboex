## Resumo Executivo

O BlackBoex implementa com sucesso os fluxos principais: criar conta, gerar API via LLM, editar codigo, testar, publicar, gerenciar billing e administrar a plataforma. A arquitetura e solida, com RBAC, multi-tenancy, audit logging, e versionamento funcionais.

A auditoria identificou problemas criticos de billing enforcement e falhas de isolamento em runtime. **Todos os problemas foram corrigidos em 2026-03-20:** billing enforcement em todos os fluxos LLM, timeout/sandbox em APIs dinamicas, streaming na geracao de codigo, usage tracking completo, BeamMonitor, e testes de integracao para fluxos criticos. Todas as 10 fases estao completas.

## Problemas Criticos (MUST FIX)

### C1. Billing enforcement bypassado na geracao de codigo LLM

**Arquivo:** `apps/blackboex_web/lib/blackboex_web/live/api_live/new.ex:219`

```elixir
# ATUAL - organization_id NAO passado
task = Task.async(fn -> Pipeline.generate(description, user_id: scope.user.id) end)
```

`Pipeline.check_llm_limit/1` tenta fazer `Keyword.fetch(opts, :organization_id)` que retorna `:error`, e o `else _ -> :ok` libera a chamada sem verificar limites.

**Impacto:** Usuario no plano free pode gerar codigo LLM ilimitadamente.

**Fix:** Passar `organization_id: scope.organization.id` nas opts.

---

### C2. Nenhum billing enforcement em chat edits (LLM)

**Arquivo:** `apps/blackboex_web/lib/blackboex_web/live/api_live/edit.ex:1256`

```elixir
# do_chat_request chama LLM diretamente sem verificar billing
case Config.client().generate_text(prompt, system: EditPrompts.system_prompt()) do
```

**Impacto:** Edicao conversacional via chat nao tem nenhum controle de plano. Uso ilimitado de tokens LLM.

**Fix:** Adicionar `Enforcement.check_limit(org, :llm_generation)` antes da chamada LLM em `do_chat_request/3`.

---

### C3. Nenhum billing enforcement na geracao de testes

**Arquivo:** `apps/blackboex_web/lib/blackboex_web/live/api_live/edit.ex:1191-1195`

```elixir
  api = socket.assigns.api
  task = Task.async(fn -> TestGenerator.generate_tests(api) end)  # Sem billing check
```

**Impacto:** Geracao ilimitada de testes via LLM.

**Fix:** Verificar `Enforcement.check_limit(org, :llm_generation)` antes do Task.async.

### C4. Nenhum billing enforcement na geracao de docs

**Arquivo:** `apps/blackboex_web/lib/blackboex_web/live/api_live/edit.ex:1231-1235`

Mesmo padrao que C3. `DocGenerator.generate(api)` chama LLM sem verificar plano.

**Fix:** Mesmo padrao — verificar enforcement antes.

---

### C5. Sem timeout na execucao de APIs dinamicas

**Arquivo:** `apps/blackboex_web/lib/blackboex_web/plugs/dynamic_api_router.ex:224-239`

```elixir
defp execute_module(conn, module, rest) do
  old_heap = Process.flag(:max_heap_size, %{size: 10_000_000, kill: false, error_logger: true})
  try do
    plug_opts = module.init([])
    module.call(conn, plug_opts)  # PODE RODAR FOREVER
  rescue ...
  after
    Process.flag(:max_heap_size, old_heap)
  end
end
```

**Problemas:**
1. **Sem timeout** — loop infinito no handler do usuario trava a conexao HTTP
2. **`kill: false`** — heap limit de 10MB so loga, nao mata o processo
3. **Sandbox module existe mas NAO e usado** no pipeline de producao

**Impacto:** DoS via API maliciosa (loop infinito ou memory bomb).

**Fix:** Usar `Task.Supervisor.async_nolink` com `Task.yield(task, 30_000) || Task.shutdown(task, :brutal_kill)`, ou integrar o `Sandbox` module que ja existe com timeout e `kill: true`.

## Problemas Importantes (SHOULD FIX)

### I1. Usage tracking incompleto para LLM

**Arquivo:** `apps/blackboex_web/lib/blackboex_web/live/api_live/new.ex:239-259`

```elixir
LLM.record_usage(%{
  output_tokens: 0,   # Hardcoded — nao rastreia tokens de saida
  cost_cents: 0,      # Hardcoded — nao calcula custo
})
```

Alem disso, o chat edit (edit.ex:1256) **descarta completamente** os dados de usage retornados pelo LLM client. Nenhum `record_usage` e chamado apos chat edits.

**Impacto:** Dashboard de billing e analytics mostram dados incompletos. Impossivel saber custo real.

**Fix:** Extrair `usage` do response do LLM client e gravar em todos os fluxos (generate, chat, tests, docs).

---

### I2. Code generation nao faz streaming para a UI

**Plano original (Fase 02, Secao 6):** "Streaming para LiveView" — deveria mostrar tokens em tempo real.

**Realidade:** `Pipeline.generate/2` chama `client.generate_text/2` (sincrono, nao streaming). A UI mostra apenas um spinner ate o resultado completo chegar.

Um `StreamHandler` module existe no codigo mas NAO e usado no fluxo de criacao.

**Impacto UX:** Usuario espera 10-30s olhando um spinner sem feedback. Experiencia degradada.

**Fix:** Usar `client.stream_text/2` + `StreamHandler` + `send()` para enviar chunks ao LiveView progressivamente.

---

### I3. Distribuicao de rate limiting e single-node

**Arquivo:** `apps/blackboex_web/lib/blackboex_web/plugs/rate_limiter.ex`

Rate limiting usa Hammer com backend ETS (in-memory). Em deploy multi-node, cada node tem seus proprios contadores.

**Impacto:** Limites efetivos multiplicam por numero de nodes.

**Fix:** Backend Redis para Hammer, ou sticky sessions no load balancer. Aceitavel para MVP single-node.

---

### I4. Testes de integracao ausentes para fluxos criticos do edit page

O `edit_test.exs` cobre mount, save, compile e rollback. **Nao cobre:**
- Fluxo completo de chat: send → LLM → diff → accept → version created
- Geracao e execucao de testes
- Publicacao completa (compile → publish → key created)
- Ciclo de vida de API keys (create → rotate → revoke)
- Billing enforcement em qualquer operacao

**Impacto:** Regressoes nesses fluxos nao serao detectadas.

**Fix:** Adicionar testes de integracao para cada fluxo critico.

---

### I5. Error handling do 3-tuple de billing

**Arquivo:** `apps/blackboex_web/lib/blackboex_web/live/api_live/new.ex:187-195`

Quando `Pipeline.generate/2` retorna `{:error, :limit_exceeded, %{plan: :free, limit: 50}}` (3-tuple), o handler trata como 2-tuple e perde os detalhes:

```elixir
def handle_info({ref, {:error, reason}}, ...) do
  # reason = :limit_exceeded — detalhes do map perdidos
  assign(socket, error: "Generation failed: #{reason}")
```

**Impacto:** Usuario ve "Generation failed: limit_exceeded" em vez de mensagem clara com link para upgrade.

**Fix:** Pattern match no 3-tuple separadamente e mostrar mensagem especifica com link `/billing`.

## Seguranca — Achados Positivos

- **IDOR protection:** Todos os recursos verificam ownership via org_id pattern match
- **AST validation:** Modulos perigosos (File, System, Process, Code, IO) bloqueados
- **Atom exhaustion:** Limite de 500 atoms no AST validator
- **XSS:** HEEx auto-escapa; nenhum uso de `raw()`
- **CSRF:** Tokens em todos os forms
- **Timing attacks:** `Plug.Crypto.secure_compare` em API keys
- **SQL injection:** Ecto parameteriza todas as queries
- **Race conditions:** SELECT FOR UPDATE em conversas e JSONB arrays
- **API key storage:** SHA-256 hash, so prefix armazenado
- **Sensitive headers:** Redacted em test history
- **Input validation:** validate_length em todos os campos string
- **Code injection em snippets:** Escaping por linguagem (shell, Python, JS, Go)
